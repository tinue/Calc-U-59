## Layer Overview

```
Calc-U-59App                 (app entry point, scene lifecycle)
└─ CalculatorView            (root view: adaptive layout, toolbar, alerts)
    ├─ KeyboardView          (calculator image + gesture hit-test + LED overlay)
    │   └─ LEDDisplayView    (7-segment Canvas renderer)
    ├─ PrinterView           (paper strip: dot-matrix or text; clipboard export)
    ├─ CardPickerView        (modal sheet: card file load/save)
    └─ DebugView             (register/SCOM/program dump panel)

EmulatorViewModel (@Observable)
├─ TI59MachineWrapper        (ObjC++ bridge object)
│   └─ TI59Machine           (C++ core, see CoreArchitecture.md)
├─ emulQueue                 (serial DispatchQueue — CPU thread)
└─ displayTimer              (60 Hz main-thread Timer)

Support files:
├─ MachineModel.swift        (enum: TI-59 / TI-58 / TI-58C variant)
├─ ROMLoader.swift           (bundle asset loading, hex decode, checksum)
├─ CardStorage.swift         (iCloud / local card file storage)
├─ StateFileLoader.swift     (.ti59 file parser, BCD encoder)
└─ PC100CFont.swift          (64-char 5×7 dot-matrix font for printer)
```

## Threading Model

| Thread | What runs there |
|---|---|
| **Main thread** | SwiftUI rendering, `tick()` timer callbacks (60 Hz), all mutations to `@Observable` properties |
| **emulQueue** (serial, `.userInteractive`) | Emulation loop (`startEmulationLoop`), `emulQueue.sync {}` drains in `loadStateFile` |
| **global/utility** | `CardStorage.warmUp()` — `url(forUbiquityContainerIdentifier:)` blocks and returns nil if called on the main thread |

`TI59Machine`'s `m_keyMutex` makes C++ state thread-safe; the Swift layer does
not need its own lock.  Display snapshots and printer output each have their own
finer-grained mutexes in C++ so the emulation thread and `tick()` don't block
each other unnecessarily.

## Emulation Loop and Timing

```
Target:  14,218.75 instructions/sec  (455 kHz ÷ 2 ÷ 16, same as hardware)
Batch:   ~284 cycles every 20 ms     (targetHz × batchMs)
```

`startEmulationLoop` runs a `while isRunning` loop on `emulQueue`:

1. Call `step()` repeatedly, accumulating the returned cycle weights.
2. When accumulated weight ≥ 284 cycles, subtract 284 (carry over overshoot to
   the next batch rather than resetting to zero — this prevents long-term drift).
3. Sleep for the remainder of the 20 ms window.

`step()` returns its cycle weight in bits 30:0 and sets bit 31 if a breakpoint
was hit.  On a breakpoint hit the loop sets `isRunning = false` and posts a
`onBreakpointHit` call to the main thread.

## 60 Hz Display Refresh (tick)

`startDisplayRefresh` creates a `Timer` on the main run loop.  Every 16 ms:

1. **Auto-eject detection**: if `cardState == .swiping` but `isCardPresent` is
   now false, the C++ layer has executed `CRD_OFF`; call `ejectCard()` to save
   any written card data.
2. **Printer drain**: collect text lines and raw code lines from the C++ printer
   queue into `printerLines` / `printerCodeLines`.
3. **Trace drain**: if tracing is enabled, pull up to 512 `TITraceEvent`s and
   append to `traceEvents` (capped at a sliding window of 512).
4. **Display update**: call `getDisplay()`, copy into local arrays, and assign
   to `@Observable` properties only when the value actually changed — guarding
   assignments prevents unnecessary SwiftUI re-renders on every tick.

## Key Input Coordinate Systems

Three coordinate spaces are involved; each layer translates between them:

| Layer | Coordinate meaning |
|---|---|
| `KeyboardView` | Visual row 0–8, col 0–4 (0-indexed from top-left of image) |
| `EmulatorViewModel.pressKey(row:col:)` | Adds 1 to each: `pressMatrixKey((row+1)*10 + (col+1))` |
| `TI59MachineWrapper.pressMatrixKey(_:)` | Matrix code = row×10 + col (range 11–95); decodes back to row/col |
| `TI59Machine.pressKey(kbit, digitSlot)` | kbit = K-line bit index (kbits[] table); digitSlot = digit-counter column |
| `TMC0501.key[]` | `key[digitSlot] \|= (1 << kbit)` — the final hardware bitmask |

The `kbits` table in `TI59MachineWrapper.mm` maps physical column (1–5) to
K-line bit index: `{0, 1, 2, 3, 5, 6}` (skipping bit 4 which is the
card-detect line, and bit 0 which is unused — hardware layout).

## Card Reader Flow

```
1. User presses "2nd Read" or "2nd Write" on the calculator keyboard.
2. ROM enters a TST BUSY polling loop; C++ sets isWaitingForCard = true.
3. The UI shows the card picker (load) or inserts a blank card (write).
4. EmulatorViewModel.insertCard(from:) or insertBlankCard(savingTo:) calls
   machine.insertCard(data) — this releases the card-switch key in C++,
   waking the ROM from its wait loop.
5. ROM executes CRD_READ + IN CRD (read) or CRD_WRITE + OUT CRD (write).
6. At the end of the pass, ROM executes CRD_OFF — C++ auto-ejects the card
   (sets isCardPresent = false, restores card-switch key).
7. tick() detects isCardPresent == false while cardState == .swiping and
   calls ejectCard(), which collects any written bytes and saves to disk.
```

Card file format (`.U59`):
- Header: 13 bytes `Calc-U-59-CRD` (ASCII)
- Payload: 246 bytes (one bank, exactly what `OUT CRD` produced)
- Legacy files (246 or 984 bytes, no header) are still accepted on load.

## State File Loading (`.ti59` / `.ti58`)

`loadStateFile` performs a **non-interactive load** — it writes state directly
to RAM without simulating key presses (except for the optional `KEYSTROKES:` section):

```
1. parseStateFile() → LoadStateResult (partition, program steps, registers, keystrokes)
2. Stop the emulation loop; drain emulQueue with sync{} to ensure no in-flight step.
3. machine.reset()
4. emulQueue.sync { m.stepN(300_000) }  ← run ROM's power-on startup routine to idle
5. m.partitionProgramRegs = n           ← write partition directly into SCOM
6. m.writeProgramSteps(data)            ← write BCD pairs into program RAM
7. m.writeDataRegister(regNum, nibbles:) ← write each register
8. startEmulationLoop()
9. playKeystrokes() async (if KEYSTROKES: section present)
```

Step 4 is necessary because the ROM's startup routine clears RAM and
initialises SCOM.  Skipping it leaves stale flags that confuse display and AOS.
`stepN(300_000)` is a conservative upper bound; the startup routine always
completes well within this.

The `StateFileLoader.swift` module also contains `encodeTI59BCD(_:)`, which
converts a Swift `Double` to the 16-nibble TI-59 BCD register format (see
`CoreArchitecture.md` for the layout).

## TI-58C Constant Memory

The TI-58C has CMOS RAM that retains its contents when powered off.  The app
emulates this by serialising the full 120×16-byte RAM to a file (`ti58c.mem`)
in the same iCloud/local directory as card files.

```
App launch:      loadConstantMemory() → deserialiseRAM(data)
                 (before emulation starts, so ROM sees warm-start flag)

Scene background / inactive:
                 persistConstantMemory() → serialiseRAM() → write to file
                 (triggered by scenePhase change in Calc-U-59App)

Hard reset:      delete ti58c.mem + deserialiseRAM(zeros) + reset()
                 (zeros ensure ROM's startup sees no valid warm-start marker)
```

## Bridge Layer (TI59MachineWrapper)

`TI59MachineWrapper` is a pure thin wrapper with no logic beyond type
translation.  Notable mappings:

**Display snapshot** — `TIDisplaySnapshot` mirrors `DisplaySnapshot` exactly;
field-by-field `memcpy` with `_Static_assert` size checks.

**Trace events** — `TITraceEvent` and `TICPUSnapshot` are ObjC structs with the
same field layout as the C++ originals.  They are boxed in `NSValue` for
transmission to Swift (Swift cannot hold C++ structs directly through the bridge).
The field-by-field copy in `drainTraceEventsMax:snapshots:` makes the layout
independence explicit and avoids UB from type-punning.

**Matrix key decode** — `pressMatrixKey:` decodes `code / 10` → digit-counter
slot, `code % 10` → physical column → K-line bit via `kbits[]`.

**RAM serialisation** — `serialiseRAM` returns a 1920-byte `NSData`
(120 regs × 16 nibbles); `deserialiseRAM` validates the length before writing.

## ROM Loading

`ROMLoader` reads `rom-59.hex` from the app bundle — a plain hex dump with 64
bytes per line.  After decoding, `wordsFromData` interprets each big-endian
16-bit pair as a 13-bit opcode word (masking off the top 3 bits).

Two sentinel values validate the TI-59 ROM image:
- `words[0]   == 0x0A01` — first opcode in the ROM (`CLR IDL` at address 0)
- `words[6143] == 0x1987` — last opcode (a backward branch; also encodes the
  chip revision year in its bit pattern)

The `MasterLibrary.hex` file is loaded the same way and fed to
`wrapper.loadLibrary`, populating the CPU's 5,000-byte library module buffer.

## LED Display Rendering

`LEDDisplayView` is a SwiftUI `Canvas` that draws 12 digit positions left to
right (display position 0 is the leftmost = most significant).  The CPU's
`A[2..13]` array maps to positions 11→0 (MSD→LSD as stored, reversed for
display).

Each digit is drawn with a ~8° italic slant via a `CGAffineTransform` shear.
Decimal point is drawn separately without the shear so it stays at the digit
boundary.

Active segments get two passes — a blurred glow and a sharper bloom — plus the
final opaque fill, giving the characteristic LED phosphor appearance.

Zero suppression logic mirrors the hardware display driver: leading zeros are
blanked unless the digit is immediately left of the decimal point (guarded by
`dpPos`), or the digit position has a control nibble (`ctrl ≥ 8`) that forces
display.

## Printer Rendering

`PrinterView` has two display modes toggled by the header button:

- **Dot mode** (`PrinterDotLine`): renders each character as a 5×7 dot grid
  using `pc100cFont`, scaled to physically accurate PC-100C dimensions
  (dot pitch ≈ 0.430 mm).  The geometry is derived from measurements of a real
  unit and documented in `PrinterView.swift`.

- **Text mode**: plain monospaced `Text` with `.textSelection(.enabled)` for
  copy-paste.

The **Copy** button renders the entire paper strip at 600 DPI using
`ImageRenderer` (physical size preserved, high DPI for laser printing) and
writes both `text/plain` and `image/tiff` (macOS) or `image/png` (iOS) to
the clipboard simultaneously.  The **Scissors** button copies then clears the
paper strip (`printerClearID` is incremented to reset the `Text` view identity
and drop any active selection).
